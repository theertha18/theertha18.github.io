Applied ML · Automotive

ECU Log Failure Prediction

An ML system at Mercedes-Benz that predicts the cause of ECU flashing failures from programming logs, and decides on its own whether to close a ticket or escalate it.

~90% accuracy · faster ticket closure · lower SLA
PythonScikit-learnPandasFlask
How I built it ›
The problem
Engineers were spending serious time manually analysing flashing issues, reading logs, working tickets, going back and forth with the workshop. It was slow and it ate into our SLA.
How I approached it
Most of the real work was the unglamorous part. I pulled raw workshop log data out of Tableau, then cleaned, pre-processed and engineered features with Pandas before training models in scikit-learn. A model is only as good as the features under it, so that's where I spent my time.
The decision I'm proud of
I didn't want a model that only suggests. I wanted one that acts safely, so I gated it on confidence. Above about 0.9 it attaches the supporting data through a per-tool Flask endpoint and closes the ticket. Below that, it escalates to a human the normal way. High precision where it's confident, nothing silent where it isn't.
Shipping it
Exposed it as a Flask API with a dedicated endpoint per internal diagnostic tool, built GUIs for live predictions, wrote tests and ran code reviews, then packaged everything as Python wheels deployed on a Linux server. I also mentored the interns who picked it up after me.
Backend · Industrial IoT

Embedded IoT Gateway Services

Go & Python microservices running on 1000+ Linux IoT gateways in off-highway vehicles, talking to the cloud over MQTT. I kept closing the gaps where we were "flying blind."

60% less manual release effort · closed real ops blind spots
GoPythonMQTTDockerJenkins
How I built it ›
The problem
We'd push a config to a device and the portal would say "done", but we had no idea if the device actually applied it. We'd only find out weeks later, when a customer complained the machine was misbehaving.
The feedback loop
I hooked into the filesystem with Python pathlib, grabbed the last-modified timestamps of every critical config file, and tied them to a system hash that verifies a job really completed. I piggy-backed the result onto the existing MQTT heartbeat, so devices report back under a new "configuration" category with no extra connection needed.
Turning SSH jobs into one click
When a config got stuck, fixing it meant a developer SSH-ing in or a technician on-site. I built a remote force-reconfiguration that wipes the hash file, so the server treats the next config as brand new. Triggered over MQTT from the dashboard, it turned a field visit into a single button.
Catching the last update before shutdown
Status went out on a fixed schedule, so if a device powered off first, the last state was lost. I added a channel-based trigger in Go to flush pending data immediately on critical events like ignition-off, which closed the observability gap. I also wrote a Go Wi-Fi config strategy (join, hotspot, or remove) that validates input and publishes precise MQTT payloads.
How I trusted it
TDD throughout. Table-driven tests with testify for every input case, plus integration tests against a lightweight MQTT broker so I could test the full command flow end to end. CI flagged regressions fast, which made the whole team more confident merging.
Backend · Data

Clinical Data Self-Service Portal

A FastAPI tool that lets medical researchers search, preview and export clinical study data themselves, without writing a line of SQL and without waiting on IT.

100% preview-to-export consistency · removed the IT bottleneck
PythonFastAPISQL ServerETL
How I built it ›
The problem
The SQL Server schema was highly normalized and dynamic, so researchers couldn't query it directly. Every data request meant me or someone in IT running a script by hand.
The tricky part
I wrote a Python ETL engine that dynamically pivots multi-type rows into clean tabular output based on each study's metadata, so a new study doesn't need new code. The engine adapts to its shape.
The decision that made it trustworthy
Web preview and production export run through one unified code path. That single decision means what a researcher sees in preview is byte-for-byte what they export. 100% consistency, no "the download looked different" surprises in a clinical setting.
Shipping it
Wrapped it in a FastAPI backend and automated the deployment with a custom orchestration step, so researchers self-serve through an internal portal instead of filing a ticket.
Computer Vision · Mobile

Lateral Flow Test Reader

A mobile app that reads a rapid-test cassette from a photo and turns it into a reproducible number, replacing "does that line look faint to you?" with an objective score.

Consistent scores across users, devices & lighting
FlutterC++OpenCVDart FFI
How I built it ›
The problem
Reading test cassettes by eye is subjective. Faint lines, different phone cameras and different lighting all push people toward different calls.
The architecture choice
Hybrid by design. Flutter handles the cross-platform UI and state, and a native C++/OpenCV library does the performance-critical image processing, connected through Dart FFI for low-latency calls and efficient memory use. UI flexibility on top, real image-processing horsepower underneath.
Killing the variability
A camera-guide overlay standardizes capture. The native pipeline then does a perspective transform to geometrically normalize the cassette, and CLAHE to flatten lighting differences between devices and environments.
The core algorithm
I collapse 2D strip regions into 1D intensity profiles, then use median- and variance-based statistics to separate a faint biological signal from background noise even in low contrast. JSON templates define cassette geometry and line positions, so a new test type is just a new template, with no change to the core algorithm.
ML · Signal Processing

Infant Seat Occupancy Detection

My master's thesis. A non-invasive way to detect an infant in a car seat using ultrasound and ML, where weight- and vision-based sensors tend to fail.

98.15% cross-validation accuracy · real-time inference
PythonScikit-learnSHAPSignal Processing
How I built it ›
The setup
A dashboard-mounted Red Pitaya STEMlab 125-14 with an SRF02 ultrasonic sensor capturing high-frequency reflections off the seat area. I built the whole capture-to-decision chain myself.
From raw signal to features
A modular signal-processing pipeline turns raw analog reflections into discriminative features like SINAD, peak count and spectral centroid. The feature design is what made the classes separable.
Choosing the model, and trusting it
I benchmarked Random Forest, KNN, SVM, CNN and LSTM. Random Forest won at 98.15%. Because it's a safety-critical system, I added an Isolation Forest to reject corrupted or anomalous signals, and used SHAP so every decision is explainable rather than a black box. A Streamlit GUI runs it live.
ML · Cloud

Cloud Anomaly Detection

A containerized service on Azure that learns patterns in streaming sequences and flags anomalies, with no labelled training data required.

Unsupervised · streaming · portable across environments
C#AzureDockerHTM
How I built it ›
The idea
Use Hierarchical Temporal Memory (neocortex-inspired) so the system continuously learns from a stream and flags deviations from its own learned baseline, with no pre-labelled data needed.
The cloud plumbing
The C# app is Dockerized, pushed to Azure Container Registry, and executed in Container Instances. Input sequences live in Blob Storage and are fetched at runtime. Runs are queue-triggered via an Azure Storage Queue carrying the experiment ID and input file.
Where results go
I split storage by purpose, structured metadata into Table Storage and detailed result logs into Blob, and supported dynamic re-runs by simply replacing the input file. Scalable and portable across environments by design.
Full-Stack · Agile Team Project

Service Management Portal

A web platform that connects service requests with providers. Users create a request, track its status, weigh up the offers that come back, and give a clear reason when they turn one down.

University team build · five sprints, Scrum + Kanban
DjangoPostgreSQLSpring BootAWSAgile
How I built it ›
What it is
It's one piece of a larger Provider Management Platform that links job availabilities with service providers. Our component owned the IT service requests, from the moment someone raises one through to the final accept-or-reject decision.
How it's built
Django runs the backend logic, PostgreSQL holds the data, and Spring Boot handles API deployment, all hosted on AWS. I wired up the APIs for login and registration, service requests, and master agreements so the components talk to each other cleanly.
How we worked
We ran it as a team across five sprints with Scrum and Kanban, moving from requirement gathering to UI, then API work, then deployment. Everything went through GitHub for version control and continuous integration.
What it does
Create a request with the expertise level and roles you need, track its status, evaluate the offers that come in, and leave structured feedback when you decline one, so providers actually learn why.
